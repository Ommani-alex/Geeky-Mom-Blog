How to Host a 7 Days to Die Server on a Synology NAS (And Keep Your Sanity)

Let’s be honest: getting a 7 Days to Die dedicated server running through Docker on a Synology NAS is a massive headache. It just is. But after enough trial, error, and contemplating throwing my laptop, I finally got ours up and running.

If you want to host your own private world without paying a monthly fee to rent a server, here is the exact process I used, along with a few common pitfalls to avoid.

The Loadout (Hardware Requirements)

Before you start clicking around in Docker, make sure your NAS actually has the resources to handle this. 7 Days to Die requires a significant amount of memory and space.

  • The NAS: Synology DS255+ (Affectionately named Jedi_Archives)
  • Memory: 18GB RAM. The absolute bare minimum memory you can get away with to run this game without it crashing when a blood moon hits is 8GB, but more is always better. Do not try to run this on the base 2GB or 4GB of RAM. The server will panic and crash the second you try to do anything. (If you need to upgrade, these are the exact RAM sticks I recommend and use: [Insert Affiliate Link Here])
  • Storage: The server files and world generation need an absolute minimum of 30GB of free space. I am running two 12TB IronWolf Pro drives set up as a mirror backup. This way, if one drive fails, I have full redundancy—which is vital because I use this NAS to back up a lot of family photos and videos, not just game servers.

Step 1: Prep Your Folders

If you haven't already, install the Container Manager (formerly Docker) from the Synology Package Center.

  • Open File Station and navigate to your docker folder.
  • Create a new folder named 7dtd (or whatever you want to call it). Inside that, create two more folders: server_data and steamcmd. This keeps your game saves and the server files organized. Setting these up now is crucial so you don't lose your entire world save the next time the container updates.

Step 2: Grab the Docker Image

Open Container Manager, click over to the Registry, and search for a reliable 7DtD image. I highly recommend using the vinanrra/7dtd-server image. It’s the most consistently maintained option I’ve found for Synology setups. Download the latest tag.

Step 3: Configure the Container (The Annoying Part)

When you launch the image to create the container, you have to get your volume mapping, environment variables, and port forwarding exactly right, or absolutely nothing will work.

  • Volume Mapping: Map the local server_data folder you just made on your NAS to the container's internal data path (usually listed in the image documentation as /home/steam/.local/share/7DaysToDie), and map your local steamcmd folder to the container's internal steamcmd path.
  • Port Forwarding: You have to map the container ports to your local NAS ports. Then, you must log into your actual home router's admin panel, find the local IP address of your Synology NAS, and forward these exact ports to allow external traffic. Otherwise, your friends won't be able to join!
    • 26900 (TCP/UDP)
    • 26901 (UDP)
    • 26902 (UDP)
  • Environment Variables: In the container settings, look for the Environment tab. This is where you set the foundation of the server. You will need to add or edit the variables provided by the image to allocate your memory limits and define basic setup variables (like ensuring the server updates on restart). Give the container access to as much RAM as you can spare.

Step 4: Tweak the serverconfig.xml

Start the container for the first time. It won't be fully playable yet, but running it once forces it to generate a serverconfig.xml file inside your server_data folder.

Stop the container. Important note here: I did not have access to a text editor directly inside Synology to edit this file. If you run into the same issue, here is the workaround:

  1. Download the serverconfig.xml file from File Station directly to your computer.
  2. Make a copy of the original file and save it somewhere safe before you delete or change anything, just in case something breaks.
  3. Open the downloaded file using your computer's native Notes app or Notepad.
  4. This is where you set your world size, blood moon frequency, server name, password, and difficulty.
  5. Tweak it to your liking, ensure it is still saved as an .xml file, and upload it back into the server_data folder on your NAS, overwriting the unedited version.
  6. Fire the container back up.

Step 5: Hurry Up and Wait

When you restart the container, it has to download the actual game files via SteamCMD. This takes a while. Do not panic if it takes a while.

Open the container logs and keep an eye on it. If you see it fetching data, go grab a coffee and leave it alone. If you see a wall of red error text, take a deep breath, go verify your folder permissions (the container needs read/write access to that 7dtd folder!), and restart it.

Once the log finally confirms the server has successfully started, boot up the game on your PC, search for your server name, enter your password, and start punching grass.
